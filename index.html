Why do Objects & Classes exist?

Before OOP, programs were written procedurally — long sequences of instructions, data passed around as loose variables. As programs grew, this became unmanageable. Two questions plagued every project: where is this data? and what code is allowed to touch it?

Objects solve both. An object bundles data (state) with the code that acts on it (behaviour). A class is the blueprint — the template you write once. Objects are the living instances created from that blueprint at runtime.

Mental Model
Think of a class as a cookie cutter and an object as a cookie. The cutter defines the shape. You can stamp out a thousand cookies, each identical in structure but different in their filling (state).

In Java, every object has two aspects: its state (stored in fields — the data it remembers) and its behaviour (exposed through methods — the things it can do). This mirrors the real world exactly.

The Three Parts of a Class

Fields
State

Variables that store an object's data. E.g. int age, String name

Constructor
Initializer

Special method called with new. Sets up initial state.

Methods
Behaviour

Functions that act on the object's state or interact with other objects.

Object vs Instance

These two words mean the same thing. An object is a concrete realization of a class in memory. When you write new Dog("Rex", "Labrador"), Java allocates heap memory, calls the constructor, and hands you back a reference to that memory location. That's your object.

The this keyword

Inside any method or constructor, this refers to the current object — the one the method was called on. This is critical when a parameter name shadows a field name:

public Dog(String name) {
    // 'name' here is the parameter, not the field!
    this.name = name; // this.name → the field; name → the parameter
}

Without this, the assignment name = name would assign the parameter back to itself — a completely useless no-op, and a classic beginner bug.

The Dog Class — Line by Line

public class Dog {               // defines the blueprint
    private String name;           // field: stores dog's name
    private final String breed;   // 'final': can never change after init
    private boolean hungry;        // field: dogs start hungry (true)

    public Dog(String dogName, String dogBreed) { // constructor
        name   = dogName;           // initialise field from parameter
        breed  = dogBreed;
        hungry = true;            // hard-coded initial state
    }

    public void bark() {           // method with no return value
        System.out.println(name + ": Woof!");
    }
}

Creating and using the object:

Dog myDog = new Dog("Rex", "Labrador");
myDog.bark();   // output: Rex: Woof!

Classic Beginner Mistakes

Mistake 1 — Shadowing without this
public Dog(String name) {
    name = name; // BUG: assigns parameter to itself!
    // the field 'name' is NEVER set
}

Fix: use this.name = name;

Mistake 2 — Forgetting to call new
Dog myDog;         // declares variable — but NO object exists!
myDog.bark();      // NullPointerException at runtime

Fix: Dog myDog = new Dog("Rex", "Lab");

Mistake 3 — Public fields
public String name; // anyone can do: myDog.name = null;

Always make fields private. Expose them only through methods.

Topic 02 · Foundation

Encapsulation

Control what others can see. Treat your class like a black box.

Access ModifiersGettersSetters

The Big Idea

Encapsulation is about hiding implementation details and only exposing a clean, controlled interface. Think about a TV remote: you press "volume up" — you don't need to know about capacitors and transistors inside. The internals are hidden. Only the buttons (the interface) are exposed.

In OOP this means: fields are always private. If someone outside needs to read or change them, they go through dedicated methods (getters/setters) that you control.

Access Modifiers — The Four Options

ModifierWho can accessWhen to use
privateOnly this classAlways for fields. Default choice for methods.
protectedThis class + subclasses + same packageWhen subclasses need direct field access (use sparingly)
publicEveryoneMethods that form the public API. Never for fields.
noneSame package onlyAvoid — breaks when you move classes to another package
Golden Rule
Give everything the strictest possible access. Start with private. Only make something less-private when you have a concrete reason to.

Getters & Setters — Why Bother?

It might seem pointless to have getAge() that just returns a field. But consider: later you need to store age in months instead of years. If you used a getter all along, the fix is trivial:

// Before: age in years
public int getAge() { return age; }

// After: age stored internally as months — zero callers need to change!
public int getAge() { return age / 12; }

Setters let you add validation logic:

public void setAge(int newAge) {
    if (newAge < 0) throw new IllegalArgumentException("Age can't be negative");
    age = newAge;
    recalculateFood(); // side-effect — only possible because of the setter
}
When NOT to add setters
If a field should never change after construction, use final and only add a getter. Not every field needs a setter — adding one opens a door to mutation you may not want.
Active Recall · Encapsulation
Why would you add a setter even when the logic is trivial — i.e. it just sets the field directly without any validation?
Because the setter acts as a future-proof contract. Today it's trivial. Tomorrow you might add validation, fire an Observer notification (notifyObservers()), or change how the data is internally represented. If all callers use the setter, you only change one place. If they access the field directly, you'd have to update every single caller.
🔒
Memory Hook: Encapsulation = TV remote. Internals hidden, only buttons exposed. FIELD = circuit board (private). GETTER/SETTER = button (public). You control what each button does.
Topic 03 · Hierarchy

Inheritance

Reuse code by making classes build on each other. The "is-a" relationship.

extendssuperis-a@Override

The Problem Inheritance Solves

Imagine a Lion class and a Giraffe class. Both have an energy field. Both have an eat(int nutrition) method. Copy-pasting that logic into every animal class is a nightmare: if you find a bug in eat(), you must fix it in every class. This violates the DRY principle (Don't Repeat Yourself).

Inheritance lets you extract the common parts into a parent class (superclass) and have child classes (subclasses) inherit them automatically.

Before inheritance: After inheritance: Animal (superclass) Lion Giraffe ├── energy: int energy energy └── eat(int nutrition) eat() eat() │ hunt() lookAtTree() ┌──────┴──────┐ neckLength Lion Giraffe hunt() lookAtTree() neckLength: int eat() [OVERRIDDEN]

The extends Keyword

public class Animal {
    protected int energy;  // protected: subclasses can access this

    public Animal(int energy) {
        this.energy = energy;
    }

    public void eat(int nutrition) {
        energy += nutrition;
    }
}

public class Lion extends Animal {   // Lion IS-A Animal
    public Lion(int energy) {
        super(energy);  // MUST call parent constructor first
    }

    public void hunt() {
        if (energy < 5) { System.out.println("Too tired"); return; }
        energy -= 5;
        eat(20);  // inherited from Animal!
    }
}

Overriding Methods

A subclass can replace a parent method with its own version. The Giraffe's eat() needs to also grow its neck, so it overrides:

@Override  // annotation tells compiler to check you got the signature right
public void eat(int nutrition) {
    super.eat(nutrition);  // call the original Animal.eat() first
    neckLength++;          // then add Giraffe-specific behaviour
}
Is-a vs Has-a — the most important design question
Only use inheritance when you can truthfully say "A is a kind of B". A Lion IS-A Animal ✓. An Engine is NOT-A Car (it HAS-A relationship). Forcing inheritance on has-a relationships creates brittle, over-coupled code. Use composition (a field) for has-a.
What is NOT inherited
Constructors and private members. Constructors are never inherited — you must write your own and explicitly call super() if needed.
Active Recall · Inheritance
A Professor class and a Student class both have a name and email. You create a Person superclass. The Professor has a teachCourse() method and the Student has a submitAssignment() method. Write the skeleton (no method bodies needed) for all three classes using proper inheritance.
public class Person {
    private String name;
    private String email;
    public Person(String name, String email) {
        this.name = name; this.email = email;
    }
}
public class Professor extends Person {
    public Professor(String name, String email) { super(name, email); }
    public void teachCourse() {}
}
public class Student extends Person {
    public Student(String name, String email) { super(name, email); }
    public void submitAssignment() {}
}
🧬
Memory Hook: Inheritance = genetic DNA. Child inherits parent's traits (fields/methods). But child can mutate them (@Override). Constructors are NOT in the DNA — each generation must call super() manually.
Topic 04 · Hierarchy

Polymorphism

One interface, many forms. The superpower that makes OOP flexible.

Type WeakeningDynamic Dispatchinstanceof

What is Polymorphism?

Polymorphism means "many forms." In Java, a variable declared as type Animal can hold any object that is a subtype of Animal — Lion, Giraffe, Elephant, whatever. The magic: when you call animal.eat(), Java automatically calls the correct version of eat() for the actual runtime type. This is called dynamic dispatch.

Animal mufasa = new Lion(200);     // compile-type: Animal; runtime type: Lion
Animal melman  = new Giraffe(100, 30);

// Call eat() on both — Java dispatches to the CORRECT override automatically
mufasa.eat(10);   // calls Animal.eat() — Lion doesn't override eat
melman.eat(10);  // calls Giraffe.eat() — Giraffe DOES override eat

The Power: Treating Different Things Uniformly

List<Animal> zoo = new ArrayList<>();
zoo.add(new Lion(200));
zoo.add(new Giraffe(100, 30));
zoo.add(new Elephant(500));

// Feed ALL animals — regardless of actual type
for (Animal a : zoo) {
    a.eat(10);  // correct eat() called for each — no if-statements needed!
}
The instanceof Anti-Pattern
// 🚫 BAD — this defeats the purpose of polymorphism
if (animal instanceof Lion) { animal.feedPrey(); }
else if (animal instanceof Giraffe) { animal.feedLeaves(); }
Instead, move the logic into an overridden method and let polymorphism handle it. If you catch yourself writing instanceof chains, rethink your class design.

What You Lose When You Weaken Type

Animal mufasa = new Lion(200);
mufasa.hunt();  // COMPILE ERROR — Animal has no hunt() method
                // Java only sees the declared type (Animal), not Lion

To get hunt() back, you'd need to cast: ((Lion) mufasa).hunt() — but only if you're sure it is actually a Lion at runtime.

Predict the Output
What does this print? 
class A {
    public void hello() { System.out.println("A"); }
}
class B extends A {
    @Override
    public void hello() { System.out.println("B"); }
}
A obj = new B();
obj.hello();
B

Even though the variable type is A, the actual runtime object is a B. Java's dynamic dispatch always calls the most-specific overridden version. This is the entire point of polymorphism.
🎭
Memory Hook: Polymorphism = actor playing multiple roles. The stage direction says "Animal enters" — but the actor is a Lion. When the director says "eat!", the Lion acts like a Lion, not a generic animal. Compile time sees the role (Animal). Runtime sees the actor (Lion).
Topic 05 · Hierarchy

Constructor Chaining

Multiple constructors, delegating construction, and the mandatory super() call.

this()super()Construction Order

Within a Class — this()

You can have multiple constructors (overloading). To avoid repeating initialization logic, one constructor can call another using this(...). This must be the first line.

public class Animal {
    private int age, weight;

    public Animal() {
        this(5);          // delegates to Animal(int age)
    }
    public Animal(int age) {
        this(age, 20);    // delegates to Animal(int age, int weight)
    }
    public Animal(int age, int weight) {
        this.age = age;
        this.weight = weight;
    }
}

To a Superclass — super()

When extending a class, the first line of your constructor must call the superclass constructor using super(...). Why? Because the superclass fields must be initialised before the subclass can use any of them.

The Golden Rule
Superclass is ALWAYS constructed before subclass. If you omit super(), Java calls the no-arg default constructor of the parent. If that doesn't exist → compile error.

Construction Order — Exam Favourite!

new Giraffe(9, 40) is called: 1. Giraffe(int age, int weight) begins 2. → calls super(age, weight) 3. → Animal(int age, int weight) runs first 4. → "CONSTRUCT: Animal(int age, int weight)" prints 5. → Animal constructor returns 6. Giraffe constructor body continues 7. → "CONSTRUCT: Giraffe(int age, int weight)" prints Rule: parent always finishes before child starts
Predict the Output
Given the Animal/Giraffe chain from your reader, what does new Giraffe(7) print? (Giraffe(int age) calls this(age, 20), which calls super(age,weight))
CONSTRUCT: Animal(int age, int weight) — Age: 7, Weight: 20
CONSTRUCT: Giraffe(int age, int weight)
CONSTRUCT: Giraffe(int age)

The chain: Giraffe(7) → this(7,20) → Giraffe(7,20) → super(7,20) → Animal(7,20) runs first, then unwinds back up the chain.
🏗️
Memory Hook: Constructor chaining = building a skyscraper. You MUST lay the foundation (parent) before the upper floors (child). super() is the foundation pour. Always happens first, no matter what.
Topic 06 · Abstraction

Abstract Classes

Templates that can never be instantiated directly — they enforce contracts on subclasses.

abstractNo InstantiationForced Implementation

The Problem: Generic Concepts That Shouldn't Exist Alone

An "Animal" is a concept — there's no such thing as a plain animal in the real world. Every real animal is a dog, a lion, a giraffe. Yet all animals share common properties (age, weight, gender) and common behaviours (eat, reproduce). This is precisely what abstract classes model.

An abstract class is a class that cannot be instantiated directly (new Animal() would be a compile error). It can only be used as a base to extend from.

public abstract class Animal {   // 'abstract' keyword here
    private int age;
    private double weight;
    private Gender gender;

    public Animal(int age, double weight, Gender gender) {
        this.age = age;
        this.weight = weight;
        this.gender = gender;
    }

    // Abstract method: no body, subclasses MUST implement
    public abstract void reproduce();

    // Normal method: subclasses inherit this as-is
    public int getAge() { return age; }
}

Animal a = new Animal(...); // COMPILE ERROR — Animal is abstract!
Animal a = new Lion(...);   // OK — Lion extends Animal and implements reproduce()

Abstract Methods — The Enforcement Contract

An abstract method has no body. When a concrete class extends the abstract class, it must implement every abstract method — otherwise it also becomes abstract (and can't be instantiated). This forces subclasses to provide their own specific behaviour.

Enum Bonus
Your reader uses Gender, Family, Species as enums. An enum is a type with a fixed set of named constants. Pattern: public enum Gender { MALE, FEMALE, OTHER }. Much safer than using raw Strings for categorical values — the compiler can check them.
abstract Animal age, weight, gender ← concrete fields (shared state) getAge(), getWeight() ← concrete methods (shared behaviour) reproduce() ← abstract method (NO body, MUST override) │ ┌─────┴──────┐ Lion Giraffe reproduce() → { ...lion specific... } reproduce() → { ...giraffe specific... }
Active Recall · Abstract Classes
Can an abstract class have a constructor? If so, when does it get called and why?
Yes. Abstract classes can (and often should) have constructors. They're called via super() from the concrete subclass constructor. Even though you can never do new Animal() directly, when you do new Lion(...), the Lion constructor calls super(...) which runs Animal's constructor to initialize the shared fields (age, weight, gender).
📐
Memory Hook: Abstract class = architectural blueprint that can never become a building alone. It defines what MUST be present (abstract methods) and what already exists (concrete methods/fields). Subclasses are the actual buildings built from it.
Topic 07 · Abstraction

Interfaces

Shared capabilities for completely unrelated classes. The "is-capable-of" relationship.

interfaceimplementsis-capable-of

The Problem Interfaces Solve

Sometimes completely unrelated classes need to share a capability. A Vehicle, a Dog, and a Baby can all make noise. They have nothing else in common — there's no sensible superclass. You can't say "a vehicle is a kind of noise-maker organism." But you can say "a vehicle is capable of making noise."

An interface defines a set of method signatures (no bodies) that any class can choose to implement. It's a contract: "I guarantee I have this behaviour."

public interface NoiseMaker {
    void makeNoise();  // no body — just the signature
}

public class Vehicle implements NoiseMaker {
    @Override
    public void makeNoise() {
        System.out.println("Vroom! Vroom!");
    }
}

public class Dog implements NoiseMaker {
    @Override
    public void makeNoise() {
        System.out.println("Woof!");
    }
}

Using Interfaces as Types

List<NoiseMaker> noisy = new ArrayList<>();
noisy.add(new Vehicle());
noisy.add(new Dog());
noisy.add(new Baby());

for (NoiseMaker n : noisy) {
    n.makeNoise();  // correct version called for each — polymorphism again!
}

A class can implement multiple interfaces

public class Dog implements NoiseMaker, Serializable, Comparable<Dog> {
    // must implement methods from ALL three interfaces
}

Java does not allow a class to extend multiple classes (single inheritance), but it can implement as many interfaces as needed.

Active Recall · Interfaces
Your assignment has a Door class that can be inspected. Should Inspectable be an abstract class or an interface? State the rule you used to decide.
Interface.

Use the rule: "A is a kind of B" → abstract class. "A is capable of [B]" → interface.

A Door is capable of being inspected — it is NOT "a kind of Inspectable". The is-a test fails. Therefore Inspectable is an interface, and Door implements it. Other unrelated classes (Chest, NPC) can also implement Inspectable without any inheritance relationship between them.
🔌
Memory Hook: Interface = electrical socket standard. The socket (interface) defines the shape. Any device (class) that fits the shape can be plugged in. A lamp and a phone are completely different things, but both are "pluggable." Abstract class = family. Interface = profession.

Final Mini Challenge

Design Question
Design the skeleton of a simple RPG system with:
• A Character abstract class (name, health, attack())
• A Warrior and Mage that extend it
• A Serializable-like interface called Saveable with a method save()
• Make both Warrior and Mage saveable

What is the type of each field? Which methods are abstract? Sketch the hierarchy.
public interface Saveable { void save(); }

public abstract class Character {
    private String name;
    private int health;
    public Character(String name, int health) {
        this.name=name; this.health=health;
    }
    public abstract void attack();
    public int getHealth() { return health; }
}

public class Warrior extends Character implements Saveable {
    public Warrior(String name) { super(name, 100); }
    @Override public void attack() { System.out.println("Sword slash!"); }
    @Override public void save() { System.out.println("Saving Warrior..."); }
}

public class Mage extends Character implements Saveable {
    public Mage(String name) { super(name, 60); }
    @Override public void attack() { System.out.println("Fireball!"); }
    @Override public void save() { System.out.println("Saving Mage..."); }
}
