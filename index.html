Topic 01 · Foundation

Objects & Classes

The atomic unit of OOP. Everything is built from these.

StateBehaviournewthis

Big Picture

A class is a blueprint defining what state (fields) and behaviour (methods) its objects will have. An object is a concrete instance created from that blueprint at runtime. Each object gets its own separate copy of the fields.

Mental Model
Class = cookie cutter. Object = cookie. Field = the filling. Method = what it does. The cutter defines shape; each cookie has its own filling.

The Three Parts of a Class

Fields
State

Variables: int age, String name

Constructor
Initializer

Called with new. Sets initial state.

Methods
Behaviour

Functions that act on state.

Line-by-Line Example

public class Dog {
    private String name;           // field — private always
    private final String breed;   // final: cannot change after init
    private boolean hungry;

    public Dog(String name, String breed) {
        this.name  = name;    // this.name = field, name = parameter
        this.breed = breed;
        hungry = true;
    }

    public void bark() {
        System.out.println(name + ": Woof!");
    }
}

Memory Diagram

Stack Heap ────────────── ───────────────────────────── myDog ──────────────► Dog object @ 0x4F3A │ name = "Rex" │ │ breed = "Labrador" │ │ hungry = true │ myDog is a REFERENCE (address), not the object itself.
Classic Bug — Shadowing
name = name; assigns parameter to itself. Field stays null. Fix: this.name = name;
Active Recall
Two objects: Dog a = new Dog("Rex","Lab"); Dog b = new Dog("Max","Poodle"); — a.bark() prints what? Why doesn't changing b's name affect a?
Prints Rex: Woof!. Each object has its own separate copy of every field in heap memory. a and b are independent objects at different memory addresses. Fields are instance variables — each instance holds its own copy.
🔑
Memory Hook: Class = blueprint. Object = building. Field = rooms. Method = what happens in rooms. this = "this specific building."
Topic 02 · Foundation

Encapsulation

Hide internals. Expose only a controlled interface.

privateGettersSetters

Big Picture

Encapsulation hides implementation details and exposes only what is necessary. Fields are always private. Anyone needing to read or change them goes through methods you write and control.

Mental Model
TV remote: buttons are public interface. Circuits inside are private. You control exactly what each button does — and can change internals without users knowing.

Access Modifiers

ModifierWho can accessUse when
privateThis class onlyAlways for fields. Default for methods.
protectedThis class + subclasses + packageFields subclasses genuinely need directly
publicEveryoneMethods forming the public API
noneSame package onlyAvoid — breaks when you move packages

Getters and Setters — Why They Matter

public class BankAccount {
    private double balance;

    public double getBalance() { return balance; }

    public void setBalance(double amount) {
        if (amount < 0) throw new IllegalArgumentException("Negative!");
        balance = amount;
    }
}
Future-proof
Even a trivial getter/setter is valuable. Tomorrow you might add validation, fire notifications, or change internal representation. All callers use the method — you change one place.
Active Recall
A BankAccount class has public double balance. Name two concrete problems this causes.
1. Anyone can write account.balance = -99999 — no validation possible. 2. If you later change how balance is stored internally (e.g. in cents), every single caller that accesses .balance directly must be updated. A getter isolates that change to one place.
🔒
Memory Hook: Encapsulation = vault. Only the teller (public methods) can access the vault (private fields). You define the rules.
Topic 03 · Hierarchy

Inheritance

Reuse code through is-a hierarchy. Eliminate duplication.

extendssuper@Overrideis-a

Big Picture

When multiple classes share fields and methods, extract them to a superclass. Subclasses extend it and inherit everything automatically. Fix a bug once — every child benefits. This is the DRY principle: Don't Repeat Yourself.

Before: Lion and Giraffe both have energy + eat() — duplicated After: Animal (superclass) protected energy: int eat(int nutrition) | +------+------+ Lion Giraffe hunt() neckLength: int eat() [OVERRIDDEN]

Core Syntax

public class Animal {
    protected int energy;
    public Animal(int e) { energy = e; }
    public void eat(int n) { energy += n; }
}

public class Lion extends Animal {
    public Lion(int e) { super(e); }  // MUST be first line
    public void hunt() { energy -= 5; eat(20); }
}

public class Giraffe extends Animal {
    private int neckLength;
    public Giraffe(int e, int n) { super(e); neckLength = n; }

    @Override
    public void eat(int n) {
        super.eat(n);   // run parent's eat first
        neckLength++;   // then add giraffe-specific behaviour
    }
}
Inherited?YesNo
public/protected methods & fields✓
Constructors✗ write your own, call super()
private members✗ hidden from subclass
Is-a vs Has-a
Only use inheritance when "A is a KIND OF B" is genuinely true. Lion IS-A Animal ✓. A Car is NOT-A Engine — it HAS-A engine. Use a field for has-a.
Active Recall
What is the difference between private and protected? When do you use protected?
private — only this class. Not even subclasses. protected — this class plus any subclass (and same package). Use protected when a field is shared state that subclasses genuinely need direct access to — like energy in Animal.
🪮
Memory Hook: Inheritance = DNA. Child inherits parent traits. Can mutate them (@Override). Constructors NOT in DNA — call super() manually.
Topic 04 · Hierarchy

Polymorphism

One call, many behaviours. The engine of flexible OOP.

Dynamic DispatchType Weakening

Big Picture

A variable declared as Animal can hold any subtype. When you call a method, Java automatically runs the most-specific overridden version for the actual runtime type. This is dynamic dispatch.

Animal mufasa = new Lion(200); Compile time: Java sees Animal — controls what methods you CAN call Runtime: Java sees Lion — controls which VERSION runs List<Animal> zoo = ...; for (Animal a : zoo) { a.eat(10); } // correct version for each — no if-statements!
Animal melman = new Giraffe(100, 30);
melman.eat(10);    // calls Giraffe's overridden eat() — not Animal's
melman.lookAtTree(); // COMPILE ERROR — Animal type has no lookAtTree()
The instanceof Anti-Pattern
Writing if (a instanceof Lion) {...} else if (a instanceof Giraffe) {...} defeats polymorphism entirely. Move behaviour into the class. Let dispatch handle it.
Predict the Output
class A { public void hello() { System.out.println("A"); }}
class B extends A { @Override public void hello() { System.out.println("B"); }}
A obj = new B(); obj.hello();
B — variable type is A, but runtime object is B. Dynamic dispatch always calls the most-specific overridden version. This is the entire point of polymorphism.
🎭
Memory Hook: Actor playing a role. Stage direction = "Animal enters" (declared type). Actual actor = Lion (runtime type). Director says "eat!" — Lion acts like a Lion, not a generic animal.
Topic 05 · Hierarchy

Constructor Chaining

Multiple constructors delegating to each other. Guaranteed parent-first order.

this()super()Construction Order

Within Class — this()

public class Animal {
    public Animal() { this(5); }
    public Animal(int age) { this(age, 20); }
    public Animal(int age, int weight) {
        this.age = age; this.weight = weight; // real work here
    }
}

To Superclass — super()

Iron Rule
Superclass ALWAYS constructed before subclass. super() must be first line. Cannot have both this() and super() in same constructor.

Construction Order — Exam Favourite

new Giraffe(7) is called: Giraffe(int age) starts → this(age, 20) Giraffe(int, int) starts → super(age, 20) Animal(int, int) runs FIRST — prints Giraffe(int, int) continues — prints Giraffe(int) continues — prints Rule: deepest constructor runs first, unwinds back up the chain
Predict the Output
class V {
  public V() { this("X"); System.out.println("V()"); }
  public V(String s) { System.out.println("V("+s+")"); }
}
class C extends V {
  public C() { super("Toyota"); System.out.println("C()"); }
}
new C();
V(Toyota)
C()

C() calls super("Toyota") first → V("Toyota") prints "V(Toyota)". Then C() continues, prints "C()".
🏗
Memory Hook: Skyscraper. Foundation (parent) always before upper floors (child). super() = the foundation pour.
Topic 06 · Abstraction

Abstract Classes

Templates that cannot be instantiated. Force subclasses to implement contracts.

abstractNo InstantiationEnforced Contract

Big Picture

A plain "Animal" doesn't exist — every real animal is a Lion, Giraffe, etc. An abstract class cannot be instantiated directly. It can have concrete fields/methods (shared logic) and abstract methods (no body — subclasses must implement).

public abstract class Animal {
    private int age;
    public Animal(int age) { this.age = age; }  // constructor — called via super()
    public int getAge() { return age; }          // concrete — shared by all
    public abstract void reproduce();              // no body — MUST override
}

new Animal(5);  // COMPILE ERROR — abstract
new Lion(5);   // OK — Lion implements reproduce()
Abstract class can haveYes/No
Fields✓ Yes
Constructors (called via super())✓ Yes
Concrete methods✓ Yes
Abstract methods (no body)✓ Yes
Instantiated directly with new✗ Never
Active Recall
Can an abstract class have a constructor? When does it run and why is it needed?
Yes. Even though you can't do new Animal(), when you do new Lion(...), Lion's constructor calls super(...) which runs Animal's constructor to initialise Animal's private fields. Without it there's no way to set the parent's private state.
📐
Memory Hook: Abstract class = architectural blueprint that can never be a building alone. Defines what MUST exist (abstract methods) and what already works (concrete methods/fields).
Topic 07 · Abstraction

Interfaces

Shared capabilities for unrelated classes. The is-capable-of relationship.

interfaceimplementsis-capable-of

Big Picture

A Dog, Vehicle, and Baby can all make noise but share no sensible ancestor. An interface defines a capability contract any class can adopt. A class can implement multiple interfaces (Java forbids extending multiple classes).

public interface NoiseMaker {
    void makeNoise();  // implicitly public abstract
}

public class Dog extends Animal implements NoiseMaker {
    @Override
    public void makeNoise() { System.out.println("Woof!"); }
}

public class Vehicle implements NoiseMaker {
    @Override
    public void makeNoise() { System.out.println("Vroom!"); }
}

List<NoiseMaker> list = ...; // Dog and Vehicle in same list — polymorphism!
for (NoiseMaker n : list) n.makeNoise();

Abstract Class vs Interface

FeatureAbstract ClassInterface
Fields✓ Yes✗ No (constants only)
Constructors✓ Yes✗ No
Concrete methods✓ Yes✓ default only (Java 8+)
A class can useOne (extends)Many (implements)
Relationshipis-ais-capable-of
Decision Rule
"A is a kind of B" → abstract class. "A is capable of doing B" → interface.
Active Recall
(a) Flyable — Bird, Plane, Superman. (b) Vertebrate — Fish, Amphibian, Mammal. (c) Printable — Document, Image, Receipt. Abstract class or interface for each?
(a) Interface — unrelated classes, pure capability. (b) Abstract class — genuine is-a, shared biological state. (c) Interface — unrelated classes, pure capability contract.
🔌
Memory Hook: Interface = electrical socket standard. Any matching device can plug in. Abstract class = family. Interface = professional skill.
Topic 08 · Design Quality

Cohesion & Coupling

The two measuring sticks of good software design.

High CohesionLow CouplingSRP

Cohesion — Inside a Class

Measures how closely related everything inside a single class is.

Low Cohesion — BAD
BankOperations creates accounts, sends emails, calculates loans, prints receipts. Four unrelated responsibilities. Any change in any system forces edits here.
High Cohesion — GOOD
AccountService only manages accounts. EmailService only handles email. Each class has one reason to change — the Single Responsibility Principle.

Coupling — Between Classes

Measures how much one class depends on or knows about another class's internals.

TypeDescriptionRating
ContentDirectly accesses another class's private fieldsWorst
CommonMultiple classes share the same global dataBad
ControlPasses flags to control another class's logicBad
ExternalDepends directly on external APIs/toolsModerate
DataOnly shares data through method parametersBest

The Master Rule

AspectCohesionCoupling
High is✓ GOOD✗ BAD
Low is✗ BAD✓ GOOD
GoalOne class, one purposeIndependent modules
Design hintSingle Responsibility PrincipleCommunicate through interfaces
Active Recall
StudentManager handles login, sends emails, writes to database, calculates grades. What is wrong and how do you fix it?
Low cohesion — 4 unrelated responsibilities, violates Single Responsibility Principle. Fix: split into AuthService (login), EmailService (email), DatabaseService (persistence), GradeCalculator (grades). Each now has one reason to change. This also improves coupling — each class is independent.
⚖
Memory Hook: "Cohesion HIGH is great. Coupling LOW is great." Inside: tight and focused. Between: loose and independent.
Topic 09 · Design Patterns

MVC — Model View Controller

Separate data, display, and input into three independent layers.

ArchitecturalSeparation of Concerns

Big Picture

As programs grow, data logic, display logic, and input handling tangle together. MVC separates them into three independent layers used in every web app, desktop GUI, and mobile app.

MODEL → data and business logic (BankAccount — no UI knowledge) VIEW → display only (BalanceScreen — no business logic) CONTROLLER → user input, coordinates Model and View Flow: USER → Controller → updates Model → Controller → refreshes View Bank analogy: YOU → Clerk (Controller) → Account (Model) → Receipt (View)
public class BankAccount {           // MODEL
    private double balance;
    public void deposit(double a) { balance += a; }
    public double getBalance() { return balance; }
}
public class AccountView {           // VIEW
    public void show(double b) { System.out.println("Balance: "+b); }
}
public class AccountController {    // CONTROLLER
    private BankAccount model;
    private AccountView view;
    public void deposit(double amount) {
        model.deposit(amount);
        view.show(model.getBalance());
    }
}
Active Recall
User clicks "deposit" in a banking app. Trace the exact sequence through MVC — which component does what and in what order?
1. Click → Controller receives event. 2. Controller calls model.deposit(amount) → Model updates balance. 3. Controller calls view.show(model.getBalance()) → View refreshes. Model never talks to View directly. View never talks to Model directly.
🏪
Memory Hook: Bank visit. YOU = user. CLERK = Controller. ACCOUNT = Model. RECEIPT = View.
Topic 10 · Design Patterns · Creational

Singleton Pattern

Guarantee exactly one instance. Global access point.

CreationalstaticLazy Init

Three Mechanisms

public class BankConfig {
    private static BankConfig instance;       // 1. static variable
    private BankConfig() { interestRate=0.05; } // 2. private constructor
    private double interestRate;

    public static BankConfig getInstance() {   // 3. one access point
        if (instance == null) instance = new BankConfig();
        return instance;  // always same object
    }
    public double getInterestRate() { return interestRate; }
}
BenefitsLimitations
Global access pointTightly couples code
Controlled instantiationHard to unit test — cannot mock easily
Consistent shared stateConcurrency issues
Lazy initialisationHidden dependencies
Exam Trap
Singleton does NOT increase testability — it decreases it. Correct characteristics: A (one instance), C (global access), D (hidden dependencies). B is wrong.
Active Recall
Why is the constructor private? What happens if it is public?
If public, anyone can write new BankConfig() creating unlimited instances — destroying the guarantee. Private constructor makes direct instantiation impossible from outside, forcing everyone through getInstance().
👑
Memory Hook: One head of state per country. Private constructor = can't elect your own king. getInstance() = official succession.
Topic 11 · Design Patterns · Structural

Adapter Pattern

Make incompatible interfaces work together without changing either.

StructuralWrapper

Structure

public interface KYCVerifier { boolean verify(String id); } // new interface
public class OldSystem { public boolean checkCustomer(int id){return true;} } // legacy

public class VerificationAdapter implements KYCVerifier {
    private OldSystem legacy;
    public VerificationAdapter(OldSystem l) { legacy = l; }

    @Override
    public boolean verify(String id) {
        return legacy.checkCustomer(Integer.parseInt(id)); // translate + delegate
    }
}
QuestionAnswer
Does Adapter modify legacy code?No — wraps it
What does Adapter implement?The NEW interface
What does Adapter hold?Reference to the OLD system
Also known asWrapper pattern
Exam Answer
Correct: A and C. The adapter allows incompatible interfaces to work together and implements the new interface while delegating to legacy code.
Active Recall
What is the fundamental difference in purpose between Adapter and Strategy?
Adapter = compatibility. Solves a structural mismatch — translates between incompatible existing interfaces. Strategy = flexibility. Enables swapping between different algorithms at runtime. Adapter fixes an interface problem. Strategy enables behavioural variation.
🔌
Memory Hook: Travel plug adapter. Your laptop (new system) plugs into the adapter. Adapter plugs into foreign socket (legacy). Neither changes.
Topic 12 · Design Patterns · Behavioral

Strategy Pattern

Swappable algorithms. No if-else chains. Change behaviour at runtime.

BehavioralComposition
public interface InterestCalculator { double calculate(double balance); }

public class SavingsInterest implements InterestCalculator {
    @Override public double calculate(double b) { return b * 0.03; }
}
public class FixedInterest implements InterestCalculator {
    @Override public double calculate(double b) { return b * 0.06; }
}

public class BankAccount {                  // context — uses strategy
    private double balance;
    private InterestCalculator calc;
    public BankAccount(double bal, InterestCalculator c) { balance=bal; calc=c; }
    public double applyInterest() { return calc.calculate(balance); }
}

// Usage — swap strategy by injecting different implementation
BankAccount savings = new BankAccount(1000, new SavingsInterest()); // 30.0
BankAccount fixed   = new BankAccount(1000, new FixedInterest());   // 60.0
Benefits
Add new strategies without changing existing code. No if-else chains. Change behaviour at runtime. Promotes composition over inheritance.
Active Recall
SortingEngine must sort differently for small vs large datasets. Name the pattern and describe the structure.
Strategy. SortingStrategy interface with sort(). Concrete strategies: SmallDataSort and LargeDataSort. SortingEngine holds a SortingStrategy field, delegates to it. Inject whichever fits at runtime.
♞
Memory Hook: Strategy = interchangeable chess pieces. Player (context) picks which piece (strategy) to use. Swap without changing the player or board.
Topic 13 · Design Patterns · Structural

Decorator Pattern

Add behaviour to individual objects. Stack wrappers freely at runtime.

StructuralWrappingDynamic

Big Picture

Book €20 + wrapping +€1 + card +€1 + VAT 21%. With inheritance you'd need one class per combination — it explodes. Decorator wraps objects inside other objects. Each wrapper adds one feature. Stack any combination at runtime.

public interface ProductPrice { double getPrice(); }

public class Book implements ProductPrice {
    @Override public double getPrice() { return 20.0; }
}

public abstract class DecoratorPrice implements ProductPrice {
    protected ProductPrice wrapped;
    public DecoratorPrice(ProductPrice w) { wrapped = w; }
}

public class WrappingPrice extends DecoratorPrice {
    public WrappingPrice(ProductPrice w) { super(w); }
    @Override public double getPrice() { return wrapped.getPrice() + 1.0; }
}
public class VATPrice extends DecoratorPrice {
    public VATPrice(ProductPrice w) { super(w); }
    @Override public double getPrice() { return wrapped.getPrice() * 1.21; }
}

// Book + Wrapping + VAT = (20+1)*1.21 = 25.41
ProductPrice order = new VATPrice(new WrappingPrice(new Book()));
System.out.println(order.getPrice());  // 25.41
InheritanceDecorator
Applies toALL instancesIndividual objects
CombinationsExplodesStack freely
Decided atCompile timeRuntime
Predict the Output
new VATPrice(new WrappingPrice(new Book())).getPrice() — show working. Book=€20, Wrap=+€1, VAT=21%.
Book=20.0 → WrappingPrice=20+1=21.0 → VATPrice=21×1.21=25.41
🥁
Memory Hook: Getting dressed. Base layer (real object) + jumper (decorator 1) + coat (decorator 2). Each wraps the previous. Remove layers without changing the person underneath.
Topic 14 · Design Patterns · Behavioral

Observer Pattern

One-to-many notification. Subscribe, unsubscribe, react automatically.

BehavioralSubjectObserver
public interface CartObserver {
    void productAdded(String p);
    void productRemoved(String p);
}

public class ShoppingCart {             // SUBJECT
    private List<CartObserver> observers = new ArrayList<>();
    public void addObserver(CartObserver o) { observers.add(o); }
    public void removeObserver(CartObserver o) { observers.remove(o); }
    public void addProduct(String p) {
        // add to list...
        for (CartObserver o : observers) o.productAdded(p);
    }
}

public class EmailNotification implements CartObserver {
    @Override public void productAdded(String p) {
        System.out.println("Email: " + p + " added.");
    }
    @Override public void productRemoved(String p) {}
}
Why This Reduces Coupling
ShoppingCart knows nothing about EmailNotification. Adding new observer = write class + register. Zero changes to ShoppingCart.
Active Recall
Decorator and Observer both use interfaces. What is the fundamental difference in purpose?
Decorator = add behaviour to an object by wrapping it. Augments method calls. Same interface as wrapped object. Example: adding VAT to price.
Observer = notify dependents when state changes. Reacts to events. Separate Observer interface. Example: notify email when cart changes.
📰
Memory Hook: Newspaper subscription. Paper (Subject) doesn't know readers personally. Just prints and delivers to all subscribers. Readers subscribe/unsubscribe freely.
Topic 15 · Design Patterns · Behavioral

Command Pattern

Encapsulate actions as objects. Replace switch with HashMap.

BehavioralHashMapDecoupling
public interface Command { void execute(Animal animal); }

public class ReproduceCommand implements Command {
    @Override
    public void execute(Animal a) { a.reproduce(); System.out.println("Reproduced."); }
}

public class CommandHandler {
    private Map<Integer,Command> map = new HashMap<>();

    public CommandHandler() {
        map.put(1, new AgeCommand());
        map.put(2, new RenameCommand());
        map.put(3, new ReproduceCommand());
    }

    public void execute(int input, Animal a) {
        Command cmd = map.get(input);
        if (cmd == null) cmd = new IdleCommand(); // always null-check!
        cmd.execute(a);
    }
}
HashMap vs Switch
O(1) lookup regardless of command count. Adding a command = one put(). No modification to execute() logic ever needed.
Active Recall
What does "decoupling sender from receiver" mean in Command? Give a concrete example.
The sender (CommandHandler) only calls execute() — it has no idea what that does. The receiver (Animal) doesn't know who triggered the action. Example: execute(3, lion) — the handler doesn't know it calls lion.reproduce(). That knowledge lives inside ReproduceCommand only.
📋
Memory Hook: Restaurant order slip. You (sender) write what you want. Waiter carries it. Chef (receiver) reads and acts. Nobody needs to know each other.
Topic 16 · Java Mechanics

Exceptions

Handle runtime errors gracefully. Never crash silently.

try-catchfinallythrowsCustom Exceptions

Big Picture

An exception is an object that represents a runtime error. Java has a structured system to catch, handle, or pass them up the call chain. Exceptions are always objects — instances of classes inheriting from Exception.

Throwable +-- Error (JVM problems — never catch these) +-- Exception +-- RuntimeException (unchecked: NullPointer, ArrayIndexOutOfBounds...) +-- Checked Exceptions (must catch or declare: IOException, FileNotFoundException...)

try-catch-finally

try {
    int[] arr = new int[2];
    System.out.println(arr[5]);       // throws ArrayIndexOutOfBoundsException
    System.out.println("after");      // NEVER reached
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("Caught: " + e.getMessage());
} catch (NullPointerException e) {     // specific BEFORE general
    System.out.println("Null!");
} finally {
    System.out.println("Always runs"); // even if exception uncaught, even on return
}
System.out.println("Program continues");

throws — Pass It Up

public readFile(String path) throws FileNotFoundException {
    // don't catch here — let the caller decide how to handle it
}

Custom Exceptions

public class InsufficientFundsException extends Exception {
    public InsufficientFundsException(String msg) { super(msg); }
}

public void withdraw(double amount) throws InsufficientFundsException {
    if (amount > balance)
        throw new InsufficientFundsException("Only " + balance + " available");
    balance -= amount;
}

try-with-resources

// resource declared in try() — closed automatically when block ends
try (Scanner sc = new Scanner(new File("data.txt"))) {
    System.out.println(sc.nextLine());
} catch (FileNotFoundException e) {
    System.out.println("File not found!");
} // sc.close() called automatically
Common Mistakes
1. Empty catch block — silently swallows errors. Never do this.
2. Catching Exception too broadly — lose information about what went wrong.
3. Wrong catch order — specific exceptions must come BEFORE general ones.
4. Catching where you can't handle it — use throws and let the right layer decide.
Predict the Output
try {
    System.out.println("A");
    int x = 10 / 0;
    System.out.println("B");
} catch (ArithmeticException e) {
    System.out.println("C");
} finally {
    System.out.println("D");
}
System.out.println("E");
A → C → D → E

"A" prints. 10/0 throws ArithmeticException. "B" skipped. Catch prints "C". Finally always runs — prints "D". Program continues — prints "E".
Active Recall
When should you use throws on a method rather than catching the exception inside it?
When the method doesn't have enough context to decide how to handle the error — the caller knows better. E.g. a file-reading utility doesn't know if a missing file should be ignored, retried, or shown as an error to the user. Declare throws and let the caller decide. Only catch where you can actually do something useful.
🛡
Memory Hook: try = risky action. catch = safety net for specific problems. finally = cleanup crew (always shows up). throws = passing the problem to whoever called you.
Topic 17 · Java Mechanics

Serialization

Save objects to disk. Reconstruct them later. Persistent state across sessions.

SerializabletransientserialVersionUID

Big Picture

When your program ends, all objects in memory are gone. Serialization converts an object to bytes that can be saved to a file or sent over a network. Deserialization reconstructs the object from those bytes. The class must implement Serializable — a marker interface (no methods, just opt-in signal).

import java.io.Serializable;

public class Vehicle implements Serializable {
    private static final long serialVersionUID = 1L;  // version identifier
    private String brand;
    private int year;
    private transient String sessionToken; // NOT saved — excluded from stream
}

Writing and Reading Objects

// SAVE to file
try (ObjectOutputStream out = new ObjectOutputStream(
        new FileOutputStream("vehicle.ser"))) {
    out.writeObject(myCar);
}

// LOAD from file
try (ObjectInputStream in = new ObjectInputStream(
        new FileInputStream("vehicle.ser"))) {
    Vehicle car = (Vehicle) in.readObject();
}

Key Concepts

ConceptWhat it does
SerializableMarker interface — opts the class into serialization
transientExcludes a field — reset to default value on load
serialVersionUIDVersion ID — prevents incompatibility when class changes
Deep serializationAll nested object fields must also implement Serializable
When to use transient
Sensitive data (passwords, tokens), temporary/cached values that can be recalculated, fields whose type is not Serializable, UI state that should reset.
Active Recall
BankAccount has: accountNumber (String), balance (double), pin (String), cachedInterest (double, recalculated each session). Which fields should be transient and why?
pin → transient. Sensitive — never save passwords/PINs as plain bytes to disk.
cachedInterest → transient. Derived/temporary — it is recalculated from balance when the object loads. No reason to persist it.
Concept Check
What is a marker interface? Why is Serializable a good example of one?
A marker interface has no methods — it is purely an opt-in signal that tells the JVM (or framework) something about the class. Serializable has no methods. Implementing it simply marks the class as "safe to serialize." The JVM checks instanceof Serializable at runtime and throws NotSerializableException if the class hasn't opted in.
💾
Memory Hook: Serialization = pressing Save on a video game. Your character's state is written to disk. Load it later — everything restored. Transient fields (like a temporary buff) reset to default.

Exam-Style Questions

Q1 — Design Scenario
When an order is placed, a warehouse is notified, a confirmation email is sent, and finance logs it. New notification types may be added. Which pattern? Name: Subject, Observer interface, 3 concrete observers. Why is this better than calling each service directly from placeOrder()?
Observer pattern. Subject: OrderSystem. Observer interface: OrderObserver with orderPlaced(Order o). Concrete observers: WarehouseNotifier, EmailService, FinanceLogger.

Better because: OrderSystem doesn't know about any specific service — low coupling. Adding a new notification = write new class + register. Zero changes to OrderSystem. Direct calls = high coupling, must modify placeOrder() every time a new service is added.
Q2 — Predict Output
abstract class Vehicle {
    protected String brand;
    public Vehicle(String b) { brand=b; System.out.println("Vehicle: "+b); }
    public abstract void move();
}
class Car extends Vehicle {
    public Car(String b) { super(b); System.out.println("Car: "+b); }
    @Override public void move() { System.out.println(brand+" drives"); }
}
class Boat extends Vehicle {
    public Boat(String b) { super(b); System.out.println("Boat: "+b); }
    @Override public void move() { System.out.println(brand+" sails"); }
    public void dock() { System.out.println(brand+" docking"); }
}
Vehicle v1 = new Car("Toyota");
Vehicle v2 = new Boat("Yamaha");
v1.move(); v2.move(); v2.dock();
Vehicle: Toyota → Car: Toyota → Vehicle: Yamaha → Boat: Yamaha → Toyota drives → Yamaha sails → COMPILE ERROR on v2.dock()

v2 is declared as type Vehicle. Vehicle has no dock() method. Declared type controls what methods you can call at compile time.
Q3 — Spot All Design Problems
public class OrderSystem {
    public static OrderSystem instance;   // Singleton attempt
    public void placeOrder(String item) {
        System.out.println("Order: "+item);
        sendEmail(item); updateInventory(item); logTransaction(item);
    }
    private void sendEmail(String i) { }
    private void updateInventory(String i) { }
    private void logTransaction(String i) { }
}
Find every problem. Name the fix for each.
1. Singleton broken — instance is public. Anyone can do OrderSystem.instance = null. Fix: make it private, add private constructor + getInstance().

2. Low cohesion — OrderSystem handles ordering, email, inventory, and logging. 4 responsibilities. Fix: extract into EmailService, InventoryService, TransactionLogger.

3. High coupling — placeOrder() directly calls each service. Adding a new notification requires modifying this class. Fix: use Observer — register services as observers, notify them on order.
Q4 — Pattern Identification
(a) Logging system — exactly one global instance. (b) Music player switching shuffle/repeat/sequential. (c) GUI app separating data, display, input. (d) Multiple services react when order placed. (e) Coffee: add milk, sugar, syrup individually to any drink.
(a) Singleton — one instance globally.
(b) Strategy — swappable algorithm.
(c) MVC — separation of concerns.
(d) Observer — one-to-many notification.
(e) Decorator — add behaviour to individual objects dynamically.
Q5 — Constructor Chaining Output
class A {
    public A() { this(1); System.out.println("A()"); }
    public A(int x) { System.out.println("A("+x+")"); }
}
class B extends A {
    public B() { System.out.println("B()"); }
}
new B();
A(1) → A() → B()

B() has no explicit super() so Java inserts super() automatically → A() starts → A() calls this(1) → A(1) runs, prints "A(1)" → A() continues, prints "A()" → B() continues, prints "B()".
Q6 — Exceptions
You call a method that can throw InsufficientFundsException. You want to always log the attempt regardless of outcome, and only show an error message if it fails. Write the try-catch-finally skeleton.
try {
    account.withdraw(500);
} catch (InsufficientFundsException e) {
    System.out.println("Error: " + e.getMessage());
} finally {
    System.out.println("Attempt logged."); // always runs
}
